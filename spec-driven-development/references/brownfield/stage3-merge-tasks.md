# 阶段3: 增量任务拆解（Brownfield）

## 目标

把本次增量需求、受影响模块和技术回写要求，更新为 Agent 可连续执行的版本化 TODO 清单。

本阶段的核心产物：

- `Documentation/DevLogs/v2.0_执行跟踪.md`
- `Documentation/DevLogs/v2.0_TODO_01.md`
- 如任务超过 30 条，继续生成：
  - `Documentation/DevLogs/v2.0_TODO_02.md`
  - `Documentation/DevLogs/v2.0_TODO_03.md`

## 核心原则

### 只拆本次变更范围

- 不重写历史任务
- 只补本次新增任务、受影响任务和回归任务

### 业务范围优先

任务必须先覆盖：

- 本次新增业务目标
- 受影响旧流程
- 受影响验收标准

然后再覆盖：

- 数据建模
- 技术架构
- Contract / 状态
- 验证与回写

### TODO 必须可连续执行

每条任务必须：

- 有明确顺序
- 有明确前置
- 有明确输出
- 有明确验收
- 能被 Agent 独立推进

## 增量任务拆解规则

### 1. 每条任务必须挂钩模块双文档

至少明确：

- 受影响模块需求文档
- 受影响模块技术文档
- 验收标准
- 是否命中通用能力
- 是否命中问题方案

### 2. 必须显式标注受影响旧流程

如果本次迭代会影响已有流程，任务中必须写出来，避免只写新增逻辑。

### 3. 禁止交叉依赖

任务依赖必须满足：

- 只能依赖更早任务
- 不允许依赖未来任务
- 不允许循环依赖
- 不允许把多个模块改动写成互相卡住的大任务组

### 4. 每个 TODO 文件最多 30 条任务

- 超过 30 条必须拆分
- 保持编号连续
- 保持顺序明确

## 推荐拆解结构

### Phase 01：旧文档与旧流程修正
- 更新受影响模块需求文档
- 更新受影响模块技术文档
- 确认旧边界变化

### Phase 02：数据 / 架构 / 契约增量
- 数据结构调整
- 状态流转调整
- Contract / API / 事件调整

### Phase 03：核心功能实现
- 新增逻辑
- 受影响逻辑修正
- 关键异常路径补齐

### Phase 04：回归与回写
- 回归受影响旧流程
- 回写实施现状
- 追加修订历史
- 必要时沉淀到 CommonCapabilities / ProblemSolutions

## TODO 文件建议格式

```markdown
# v2.0 TODO 01

## 说明
- 文件范围：TODO-001 ~ TODO-030
- 当前阶段：Phase 03 / 增量核心功能
- 前置文件：受影响模块需求文档、受影响模块技术文档、执行跟踪文档

## 任务清单

### TODO-001 [ ] 更新在线阅读离线化模块需求文档中的跳转边界
- 所属模块：在线阅读离线化
- 来源需求文档：在线阅读离线化_需求文档.md
- 来源技术文档：在线阅读离线化_技术文档.md
- 前置任务：无
- 输出：更新后的业务边界与验收描述
- 验收：新需求边界已在需求文档中稳定落盘

### TODO-002 [ ] 更新离线跳转状态流转设计
- 所属模块：在线阅读离线化
- 来源需求文档：在线阅读离线化_需求文档.md
- 来源技术文档：在线阅读离线化_技术文档.md
- 前置任务：TODO-001
- 输出：更新后的状态流转说明
- 验收：技术文档可支持后续实现与验证
```

## 执行跟踪更新要求

执行跟踪文档中至少要补写：

- 当前 TODO 文件
- 当前任务指针
- 本次变更范围
- 风险与阻塞
- 文档回写记录

## 完成标准

- ✅ 当前版本执行跟踪文档已更新
- ✅ 当前版本 TODO 清单已生成
- ✅ 超过 30 条任务时已自动拆分到后续 TODO 文件
- ✅ 本次任务与模块双文档形成闭环
- ✅ 任务依赖只指向更早任务
- ✅ 用户确认拆解结果后，进入阶段4
