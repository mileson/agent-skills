# 阶段1: 增量需求沟通（Brownfield）

## 目标

基于现有文档，先输出当前产品定位审计摘要，再让用户确认“这次新需求是在延伸原定位，还是改变原定位”，最后再进入增量需求沟通与文档更新。

## 强制门禁

在 Brownfield 场景下，**先理解旧文档、再确认定位变化、再更新主文档** 是硬规则。

禁止直接做以下动作：

- 直接更新 `Basic` 文档
- 直接更新模块需求文档 / 模块技术文档
- 直接生成增量 Mermaid / ASCII
- 直接拆任务

如果当前定位、本次边界或本次验收仍不清楚，不允许开始更新文档。

## 阶段 0.5：产品定位审计摘要（必须）

在正式提问前，Agent 必须先基于现有文档输出：

```text
当前产品定位（基于现有文档）
- 当前产品核心目标：
- 当前主要用户群体：
- 当前核心场景：
- 当前版本边界：
- 当前相关模块：
- 当前已知约束：

本次新需求的初步理解
- 你新提的需求是：
- 我判断它更像：
  - 定位延伸
  - 定位变更
  - 单模块增强
  - 跨模块变更

潜在冲突点
- 与现有定位一致的地方：
- 与现有定位可能冲突的地方：
- 需要你确认是否变化的地方：
```

用户必须先确认：

1. 现有定位是否仍成立
2. 这次需求是扩展还是定位变化
3. 哪些旧边界本次要打破

## 4 轮增量访谈协议

### Round 1: 定位变化确认

#### 必问内容

1. 这次需求是在原定位内增强，还是产品方向发生变化？
2. 原边界是否变化？
3. 原验收标准是否变化？
4. 原有模块职责是否要调整？

#### 输出

- 定位变化判断
- 受影响边界摘要
- 受影响模块摘要

### Round 2: 增量目标 / 边界 / 验收访谈

#### 必问内容

1. 本次迭代真正要达成什么结果？
2. 哪些现有行为不能被破坏？
3. 本次明确不做什么？
4. 本次成功的判断标准是什么？

#### 输出

- 增量目标
- 增量边界
- 增量非范围
- 增量验收口径

### Round 3: 增量场景循环访谈

至少覆盖：

- 新增场景
- 受影响的旧场景
- 异常场景
- 边界场景

#### 输出

- 场景清单
- 受影响范围
- 关键歧义点

### Round 4: 增量流程澄清

#### 每个关键流程必须覆盖

1. 前置条件
2. 触发动作
3. 主流程
4. 分支流程
5. 异常流程
6. 结束状态
7. 用户可见反馈

#### 输出

- 可直接用于 Mermaid 的增量流程定义
- 可直接用于模块需求文档更新的流程内容
- 可直接用于业务 ASCII 更新的页面 / 状态关系定义

## 每轮呈现格式（必须）

### 面向用户必须这样呈现

每轮结束时，Agent 面向用户的输出顺序固定为：

1. 当前理解摘要
2. 已确认内容
3. 仍需用户确认的点
4. 如当前轮需要，可补一个 ASCII 或 Mermaid 草图
5. 本轮问题清单

### 问题清单呈现要求

- 必须放在消息最后
- 必须使用代码块包裹
- 不要把问题清单穿插在分析中间
- 每轮最多 5 个问题
- 推荐控制在 3-5 个问题
- 如果剩余问题很多，必须按优先级拆到下一轮
- 优先问最影响定位变化、变更边界、验收口径和受影响流程的问题

**推荐格式**：

```text
请你按顺序回答下面这些问题：

1. ...
2. ...
3. ...
```

### 不要直接展示给用户的内容

以下属于 Agent 内部推进判断，不需要直接输出给用户：

- 是否允许进入下一轮
- 当前歧义等级
- 当前风险等级
- 内部放行状态

如果 Agent 判断还不能进入下一轮，应当把它转化成“请你先补充回答哪些问题”，而不是直接把内部状态标签展示给用户。

## 放行门禁

只有以下内容清楚后，才允许更新主文档：

- 当前定位已审计
- 用户已确认定位是否变化
- 本次变更范围清楚
- 受影响边界清楚
- 受影响场景清楚
- 受影响流程清楚
- 本次验收标准清楚

## 文档更新要求

### Basic 层

只更新受影响的基础文档，不重复重写全量内容。

### Modules 层

每个受影响模块都要按双文档更新：

- `<模块名>_需求文档.md`
  - 更新业务场景
  - 更新业务场景流程（Mermaid）
  - 更新业务 ASCII
  - 更新验收标准
- `<模块名>_技术文档.md`
  - 更新需求映射
  - 更新系统架构 / 数据 / Contract / 技术流程
  - 更新实施现状
  - 更新代码同步说明

### Framework 层

如本次迭代出现以下情况，需要同步沉淀：

- 多模块复用能力：更新或新增 `CommonCapabilities`
- 新踩坑 / 新反模式 / 新标准解法：更新或新增 `ProblemSolutions`

## 修订历史要求

所有受影响正式文档都必须：

- 更新 `文档版本`
- 更新 `更新日期`
- 更新 `更新人`
- 追加 `文档修订历史`

## 完成标准

满足以下条件后，阶段1完成：

- 已完成产品定位审计摘要
- 已完成 4 轮增量访谈
- 用户确认本次变更定位
- 受影响文档已更新
- 修订历史已追加

## 输出清单

- ✅ 更新受影响 `Basic/*`
- ✅ 更新受影响 `Documentation/Modules/<模块名>/<模块名>_需求文档.md`
- ✅ 更新受影响 `Documentation/Modules/<模块名>/<模块名>_技术文档.md`
- ✅ 如需要，更新 `Framework/CommonCapabilities/*`
- ✅ 如需要，更新 `Framework/ProblemSolutions/*`
