# 阶段3: 任务拆解（Greenfield）

## 目标

将已经稳定的产品业务文档与技术文档，拆解成 Agent 可连续执行的长程任务清单。

本阶段的核心产物不再只是“阶段任务列表”，而是：

- `Documentation/DevLogs/v1.0_执行跟踪.md`
- `Documentation/DevLogs/v1.0_TODO_01.md`
- 如任务超过 30 条，继续生成：
  - `Documentation/DevLogs/v1.0_TODO_02.md`
  - `Documentation/DevLogs/v1.0_TODO_03.md`

## 核心原则

1. 产品业务优先
- 任务必须先忠实覆盖业务目标、业务规则、业务场景和验收标准。
- 不允许为了技术实现方便而改变业务闭环。

2. Agent 负责技术落地
- 数据建模
- 架构分层
- 技术规格
- 技术逻辑
- 技术选型

这些内容应由 Agent 基于业务文档和技术文档继续收敛，而不是反过来要求用户补技术细节。

3. TODO 必须穷尽范围
- 所有已确认的业务范围和技术范围，最终都必须映射到 TODO 任务。
- 不能只拆“开发任务”，必须同时包含：
  - 文档回写任务
  - 验证任务
  - 必要的架构/数据/契约任务

## 任务拆解原则

### 1. 每条任务必须是最小闭环

每条任务应满足：

- 能被单独执行
- 能被单独验证
- 有明确输入
- 有明确输出
- 有明确验收结果

### 2. 每条任务必须绑定文档

每条任务至少要能回答：

1. 来源需求文档是什么
2. 对应技术文档是什么
3. 验收依据是什么
4. 是否命中通用能力
5. 是否命中问题方案

### 3. 禁止交叉依赖

任务依赖必须满足：

- 只能依赖更早编号的任务
- 禁止依赖未来任务
- 禁止循环依赖
- 禁止“任务 A 依赖任务 B，任务 B 又依赖任务 A”
- 禁止把多个大任务互相耦合成一组无法独立推进的任务

正确目标是：

- 让 Agent 可以从 `TODO-001` 一直顺着推进到 `TODO-002`
- 完成当前任务后自然推进到下一任务

### 4. 每个 TODO 文件最多 30 条任务

- `TODO_01.md`：最多 30 条
- 超过 30 条就拆到 `TODO_02.md`
- 再超过就继续拆 `TODO_03.md`

拆分时应保持：

- 同一主题或同一阶段尽量放在同一个 TODO 文件中
- 但不能为了主题完整而突破 30 条上限

## 推荐拆解顺序

建议按下面的先后顺序拆：

### Phase 01：文档与结构基线
- 明确模块边界
- 完成模块需求文档补齐
- 完成模块技术文档补齐
- 确认数据模型 / 契约 / 关键流程

### Phase 02：底层实现基线
- 数据建模
- 基础结构
- 核心能力接入
- 通用能力接入

### Phase 03：核心业务闭环
- 核心主流程
- 关键状态流转
- 关键异常处理

### Phase 04：补强与验证
- 回归测试
- 文档回写
- 修订历史追加
- 问题方案沉淀

## TODO 文件建议格式

```markdown
# v1.0 TODO 01

## 说明
- 文件范围：TODO-001 ~ TODO-030
- 当前阶段：Phase 02 / 核心实现基线
- 前置文件：模块需求文档、模块技术文档、执行跟踪文档

## 任务清单

### TODO-001 [ ] 建立离线页面实体模型
- 所属模块：在线阅读离线化
- 来源需求文档：在线阅读离线化_需求文档.md
- 来源技术文档：在线阅读离线化_技术文档.md
- 前置任务：无
- 输出：页面实体模型、字段定义、状态定义
- 验收：模型可以承载页面、资源、链接关系

### TODO-002 [ ] 建立资源去重模型
- 所属模块：在线阅读离线化
- 来源需求文档：在线阅读离线化_需求文档.md
- 来源技术文档：在线阅读离线化_技术文档.md
- 前置任务：TODO-001
- 输出：资源去重规则、资源引用结构
- 验收：同一资源不会重复存储
```

## 执行跟踪文档写法

执行跟踪文档只负责：

- 当前版本目标
- 当前 TODO 文件
- 当前任务指针
- 当前阶段状态
- 风险与阻塞
- 文档回写记录

推荐写法：

```markdown
## 当前执行定位

- 当前 TODO 文件：v1.0_TODO_01.md
- 当前任务：TODO-008
- 当前阶段：Phase 02
- 当前状态：进行中
```

## 拆解完成后的检查

至少检查以下 3 条链：

1. 模块需求文档 -> 模块技术文档：是否完整覆盖
2. 模块需求文档 -> TODO 清单：是否形成闭环任务
3. 模块技术文档 -> 修订历史 / 代码同步说明：是否预留回写动作

## 完成标准

- ✅ 已生成版本化执行跟踪文档
- ✅ 已生成至少一个 `TODO_01.md`
- ✅ 如任务超过 30 条，已继续拆分到 `TODO_02.md / TODO_03.md`
- ✅ 每条任务都绑定来源文档与验收标准
- ✅ 所有任务按顺序编号
- ✅ 所有依赖仅指向更早任务
- ✅ 用户确认拆解结果后，进入阶段4
