# 阶段1: 需求沟通（Greenfield）

## 目标

在 Greenfield 场景下，先把产品定位、用户群体、首版范围、业务场景和关键流程问清楚，再初始化 `Documentation/` 并开始落文档。

本阶段不是“一轮提问后直接写 PRD”，而是固定的 4 轮访谈协议。

## 强制门禁

在以下动作发生前，必须完成需求澄清轮次并得到用户确认：

- 初始化 `Documentation/`
- 运行 `init_project_structure.py`
- 生成任何 PRD 草稿
- 生成模块需求文档 / 模块技术文档
- 生成 Mermaid 流程图
- 生成 ASCII 交互稿

如果仍存在高歧义项，不允许进入文档生成。

## 4 轮访谈协议

### Round 1: 产品定位访谈（战略层 / 范围层 / 交互认知预判）

这轮融合《用户体验五要素》的：

- 战略层
- 范围层
- 结构层预判

如需求涉及页面、流转、交互，还必须额外应用《Don’t Make Me Think》的认知负担原则。

#### 必问内容

**战略层**
1. 产品核心目标是什么？
2. 主要用户是谁？次用户是谁？
3. 用户当前如何解决这个问题？为什么现有方案不够好？
4. 如果首版只能证明一件事，最想验证什么？

**范围层**
1. 首版必须具备哪些能力？
2. 明确不做什么？
3. 哪些是未来可能做，但本次不纳入？
4. 最小可成立闭环是什么？

**结构层预判**
1. 用户进入产品后的第一步是什么？
2. 最重要的主任务是什么？
3. 产品更像工具流、内容流还是任务流？
4. 用户最可能在哪一步迷失？

**交互认知预判（仅在涉及交互时启用）**
1. 用户打开后最重要的动作是否一眼明确？
2. 页面最重要的信息是否足够突出？
3. 功能命名是否自解释？
4. 核心路径是否过长？
5. 哪些地方最容易让用户困惑或误操作？
6. 失败后恢复路径是否清楚？

#### 输出

- 产品定位摘要
- 用户群体摘要
- 典型需求场景摘要
- 首版范围摘要
- 核心任务路径摘要
- 交互认知风险摘要（如适用）

### Round 2: 目标 / 边界 / 验收访谈

#### 必问内容

1. 这次首版真正要解决什么？
2. 哪些边界不能碰？
3. 哪些明确不做？
4. 什么叫成功？
5. 哪些结果是最低可接受标准？

#### 输出

- 首版目标
- 首版边界
- 非范围说明
- 验收口径

### Round 3: 业务场景循环访谈

至少覆盖：

- 主成功场景
- 关键分支场景
- 异常场景
- 边界场景

#### 每个场景都要问清

1. 谁在什么情况下触发？
2. 用户想完成什么？
3. 中间会看到什么反馈？
4. 可能出现什么分支或异常？
5. 最终什么状态算完成？

#### 输出

- 场景清单
- 场景说明
- 关键歧义点

### Round 4: 业务场景流程澄清

这轮用于把需求压实到可写 Mermaid、可写 ASCII、可拆任务。

#### 每个关键流程必须覆盖

1. 前置条件
2. 触发动作
3. 主流程步骤
4. 分支流程
5. 异常流程
6. 结束状态
7. 用户可见反馈

#### 输出

- 可直接用于 Mermaid 的流程定义
- 可直接用于模块需求文档的业务流程说明
- 可直接用于 ASCII 的页面 / 状态关系定义

## 每轮呈现格式（必须）

### 面向用户必须这样呈现

每轮结束时，Agent 面向用户的输出顺序固定为：

1. 当前理解摘要
2. 已确认内容
3. 仍需用户确认的点
4. 如当前轮需要，可补一个 ASCII 或 Mermaid 草图
5. 本轮问题清单

### 问题清单呈现要求

- 必须放在消息最后
- 必须使用代码块包裹
- 不要把问题清单夹在中间
- 不要把问题清单写成散落段落，避免用户漏看
- 每轮最多 5 个问题
- 推荐控制在 3-5 个问题
- 如果剩余问题很多，必须按优先级拆到下一轮
- 先问最影响产品定位、首版边界、验收口径和核心流程的问题

**推荐格式**：

```text
请你按顺序回答下面这些问题：

1. ...
2. ...
3. ...
4. ...
```

### 不要直接展示给用户的内容

以下属于 Agent 内部判断，不需要直接输出给用户：

- 是否允许进入下一轮
- 当前歧义等级
- 当前风险等级
- 进入下一轮前必须解决的问题（如只是 Agent 内部推进判断）

这些内容可用于 Agent 自己决定是否继续，但对用户只需要转化成“请你补充回答什么”。

禁止直接进入文档编写。

## 放行门禁

只有以下内容清楚后，才允许初始化 `Documentation/`：

- 产品定位清楚
- 用户群体清楚
- 典型需求场景清楚
- 首版目标清楚
- 首版边界清楚
- 验收标准清楚
- 至少 3 个关键业务场景走通
- 至少 1 个关键异常流程走通
- 如涉及交互，核心认知路径已初步澄清

## 进入文档落盘后的产出要求

### Basic 层

- `Documentation/Basic/PRD/1_产品概述.md`
  - 产品背景
  - 目标用户
  - 核心价值
  - 首版产品级 ASCII 草图
- `Documentation/Basic/PRD/2_用户画像.md`
- `Documentation/Basic/PRD/3_核心流程.md`
  - 产品级 Mermaid 流程图
- `Documentation/Basic/PRD/4_边界场景.md`

### 模块层

识别首批核心模块，并在 `Documentation/Modules/` 下建立双文档结构：

- `<模块名>_需求文档.md`
- `<模块名>_技术文档.md`

其中：

- 模块需求文档负责业务场景、业务流程（Mermaid）、功能需求、业务 ASCII、验收标准
- 模块技术文档负责需求映射、架构、数据 / 状态、Contract、技术流程、实施现状、测试设计

## 完成标准

满足以下条件后，阶段1完成：

- 4 轮访谈完成
- 门禁通过
- 用户确认需求理解无重大偏差
- 已落盘首版 Basic/PRD 文档
- 已建立首批模块双文档骨架
