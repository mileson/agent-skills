# 文档粒度控制指南

## 文档拆分原则

### 何时拆分文档

**拆分信号**:
1. 单个文档超过 500 行
2. 包含多个独立主题
3. 审查时用户难以一次性消化

### 拆分示例

#### PRD 文档细分

**原结构**:
```
Basic/PRD/
└── 1_产品概述.md（包含所有内容）
```

**细分后**:
```
Basic/PRD/
├── 1_产品概述/
│   ├── 1.1_核心价值.md
│   ├── 1.2_目标用户.md
│   ├── 1.3_竞品分析.md
│   └── 1.4_核心功能.md
├── 2_用户画像/
│   ├── 2.1_用户群体.md
│   ├── 2.2_使用场景.md
│   └── 2.3_用户需求.md
└── 3_核心流程/
    ├── 3.1_注册登录流程.md
    ├── 3.2_核心业务流程.md
    └── 3.3_异常处理流程.md
```

#### 技术方案细分

**原结构**:
```
Basic/Architecture/
└── 2_系统架构.md（包含所有架构设计）
```

**细分后**:
```
Basic/Architecture/
├── 2_系统架构/
│   ├── 2.1_分层架构.md
│   ├── 2.2_模块划分.md
│   ├── 2.3_技术选型.md
│   └── 2.4_部署架构.md
└── 3_交互时序图/
    ├── 3.1_用户注册流程.md
    ├── 3.2_订单处理流程.md
    └── 3.3_支付流程.md
```

## 命名规范

### 层级命名

**一级目录**: `Basic/PRD/`, `Basic/Architecture/`, `Basic/Database/`

**二级文件**: `1_产品概述.md`, `2_用户画像.md`

**三级目录**: `1_产品概述/`, `2_用户画像/`

**四级文件**: `1.1_核心价值.md`, `1.2_目标用户.md`, `2.1_用户群体.md`

### 编号规则

- 使用数字前缀，便于排序
- 一级目录：`01`, `02`, `03`（两位数）
- 二级文件/目录：`1`, `2`, `3`（一位数）
- 三级文件：`1.1`, `1.2`, `2.1`（小数点表示层级关系）

## 行数限制

### 推荐限制

| 文档类型 | 单文件行数 | 单目录总行数 | 处理建议 |
|---------|-----------|-------------|---------|
| **PRD** | 200-500 行 | 5000-10000 行 | 超过 500 行拆分 |
| **技术方案** | 200-500 行 | 5000-10000 行 | 超过 500 行拆分 |
| **数据库设计** | 500-1000 行 | 5000-10000 行 | 超过 1000 行拆分表 |
| **API 文档** | 200-500 行 | 5000-10000 行 | 超过 500 行按模块拆分 |

### 自动检查

使用 `scripts/check_doc_size.py` 自动检查文档行数：

```bash
python3 scripts/check_doc_size.py --doc-path Documentation/ --warn 400 --max 500
```

**参数说明**:
- `--doc-path`: 文档根目录
- `--warn`: 警告阈值（默认 400 行）
- `--max`: 最大限制（默认 500 行）

**输出示例**:
```
⚠️ 警告: Documentation/Basic/PRD/1_产品概述.md (450 行) 接近限制

❌ 错误: Documentation/Basic/Architecture/2_系统架构.md (650 行) 超过限制
   💡 建议拆分为 4 个文件
   📋 拆分策略: 按架构层次拆分（如：分层架构、模块划分、技术选型、部署架构）
   📁 目标目录: Basic/Architecture/2_系统架构/
   📝 命名规则: 2.1_xxx.md, 2.2_xxx.md, ...
```

**说明**:
- Agent 收到这个输出后，会读取 `2_系统架构.md` 的内容
- 根据实际内容决定如何命名（如 `2.1_分层架构.md`, `2.2_模块划分.md`）
- 不需要写死具体文件名，保持灵活性

## 拆分策略

### 按主题拆分

**适用场景**: 文档包含多个独立主题

**示例**: `1_产品概述.md` 拆分为：
- `1.1_核心价值.md`
- `1.2_目标用户.md`
- `1.3_竞品分析.md`
- `1.4_核心功能.md`

### 按模块拆分

**适用场景**: 文档按模块组织（如 API 文档）

**示例**: `1_RESTful_API.md` 拆分为：
- `1.1_用户模块API.md`
- `1.2_订单模块API.md`
- `1.3_支付模块API.md`

### 按表拆分

**适用场景**: 数据库设计文档过大

**示例**: `1_数据表设计.md` 拆分为：
- `1.1_用户相关表.md`（users, user_profiles, user_roles）
- `1.2_订单相关表.md`（orders, order_items, payments）
- `1.3_商品相关表.md`（products, categories, inventory）

## 最佳实践

### 审查阶段检查

在每个阶段完成后，使用自动检查工具验证：

```bash
# Stage 1: 需求沟通完成后
python3 scripts/check_doc_size.py --doc-path Documentation/Basic/PRD

# Stage 2: 架构设计完成后
python3 scripts/check_doc_size.py --doc-path Documentation/Basic/Architecture

# 全局检查
python3 scripts/check_doc_size.py --doc-path Documentation/
```

### 用户确认时提示

向用户展示文档时，提示文档行数：

```
已生成 PRD 文档（共 3 个文件，1200 行）：
- ✅ 1_产品概述.md (380 行)
- ✅ 2_用户画像.md (420 行)
- ⚠️ 3_核心流程.md (400 行，接近限制)

请审查确认，如需拆分请告知。
```

### 增量式拆分

不要一次性生成大文档，而是逐步生成：

1. 先生成主文件（如 `1_产品概述.md`）
2. 用户审查确认
3. 如超过限制，再拆分为子文件

这样可以：
- 降低用户审查负担
- 及时发现需要调整的地方
- 避免大规模重构
