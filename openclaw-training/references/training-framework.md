## 学员档案驱动原则

每次培训方案设计必须以学员档案为起点：

1. **新学员**（无档案）：第一次培训侧重全面摸底，覆盖多个能力域，建立基线数据
2. **老学员**（有档案）：
   - 针对 ⚠️ 需关注的能力设计强化任务
   - 针对 ❌ 未通过的能力设计重训任务
   - ✅ 已认证的能力不重复考核，除非相关工具/配置有变更
   - 参考上次培训的改善建议，验证建议是否已落实
3. **成长对比**：报告中必须包含与上次培训的评分对比表，体现学员的纵向成长

## 轮次设计原则

每次专题培训采用**递进式多轮结构**，难度逐步提升：

| 轮次 | 目标 | 任务类型 | 难度 |
|------|------|---------|------|
| 第一轮：基础确认 | 验证 Agent 理解了新知识/工具 | 知识复述、配置验证、简单执行 | 低 |
| 第二轮：实战考核 | 验证 Agent 能独立完成真实任务 | 完整任务流程、交付物产出 | 中 |
| 第三轮：压力测试 | 验证 Agent 在限制条件下的表现 | 限时任务、快速连续任务、复合任务 | 高 |
| 第四轮：综合考核（可选） | 端到端模拟真实工作场景 | 多工具协同、多步骤串联 | 最高 |

**轮次数量**：根据培训主题复杂度选择 2-4 轮。简单主题 2 轮即可，复杂主题建议 3-4 轮。

## 标准轮次结构

### 第一轮：基础确认（2-4 个任务）

目的：确认 Agent 已读取并理解新增的配置/知识。

典型任务模式：
- `T1-1` 知识确认：让 Agent 读取指定文件，用自己的话复述要点
- `T1-2` 工具验证：让 Agent 执行一次基础工具操作，确认工具可用
- `T1-3` 工作区创建：让 Agent 按模板创建标准工作区目录

评分重点：理解准确性、操作正确性。

### 第二轮：实战考核（2-5 个任务）

目的：验证 Agent 能独立完成完整任务流程并产出交付物。

典型任务模式：
- `T2-1` 完整流程执行：给定真实主题，执行完整工作流
- `T2-2` 交付物产出：检查产出文件的结构、内容、质量
- `T2-3` 工具链协同：需要串联多个工具完成的任务

评分重点：交付物质量、流程完整性、工具使用正确性。

### 第三轮：压力测试（2-3 个任务）

目的：验证 Agent 在约束条件下仍能保持质量。

典型任务模式：
- `T3-1` 限时快速任务：限定时间内完成简化版任务
- `T3-2` 质量自检：Agent 自查前序产出的完整性和规范性
- `T3-3` 连续快速任务：3 个短任务连续执行，考察稳定性

评分重点：速度、稳定性、自检能力。

## 任务设计规范

每个任务必须包含以下字段：

```
任务 ID：T{轮次}-{序号}（如 T1-1, T2-3）
任务名称：简短描述
目标：Agent 需要达成的具体目标
指令：发送给 Agent 的完整消息文本
交付物：期望 Agent 产出的具体文件或操作结果
评分维度：本任务的评分侧重点
满分：100
```

### 指令编写原则

1. **明确具体**：告诉 Agent 做什么，不告诉怎么做（考核其自主能力）
2. **可验证**：交付物必须是可检查的（文件、命令输出、状态变化）
3. **独立可评**：每个任务独立打分，不依赖其他任务的完成

## 培训执行规范

### 消息发送

通过 OpenClaw CLI 发送任务指令：
```bash
[连接命令] 'openclaw agent --channel [channel] --session-id "training-[date]" --message "[指令内容]" --timeout [秒数]'
```

连接命令从 `data/connections.md` 读取。

### 等待与监控

- 简单任务：`--timeout 300`（5分钟）
- 复杂任务：`--timeout 600`（10分钟）
- 发送后等待（`sleep`），然后检查 Agent 产出
- 如果超时无响应，记录为"超时未完成"并评分

### 产出物检查

通过连接命令检查 Agent 产出：
```bash
[连接命令] 'ls -la [工作区路径]/[交付物路径]'
[连接命令] 'cat [工作区路径]/[交付物文件]'
[连接命令] 'wc -l [工作区路径]/[交付物文件]'
```

## 产出物实地验证规范

> ⚠️ **这是评分的前置条件。未经教练实地验证的任务，不得进入评分环节。**
> 教训来源：2026-02-18 T2-3，Agent 报告"即刻发帖成功"但配图中文全部乱码，教练仅凭报告打了 95 分（后修正为 75 分）。

### 验证原则

1. **眼见为实**：Agent 的文字报告（"发布成功""图片已生成"）只是线索，不是证据
2. **逐项验证**：每个交付物都必须独立验证，不能因为"上一步成功了"就跳过
3. **记录留痕**：验证过程和结果必须写入 session-log

### 按产出类型的验证 SOP

#### 1. 服务器文件（Markdown / JSON / 代码等）

```bash
# 必须执行：读取文件内容
[连接命令] 'cat [文件路径]'

# 确认项：
# - 文件非空（不是空模板）
# - 内容有实质信息（不是只有标题和占位符）
# - 编码正确（无乱码）
# - 格式规范（Markdown 语法正确、JSON 可解析）
```

#### 2. 图片文件

```bash
# Step 1: 下载到本地
scp [连接参数] [远程路径] /tmp/[本地文件名]

# Step 2: 用 Read 工具查看图片（Cursor 支持直接查看图片）
# Read /tmp/[本地文件名]

# 确认项：
# - 图片可正常打开
# - 中文/英文文字渲染正确，无乱码、无方块字符
# - 图片内容与任务主题相关
# - 图片尺寸和质量合理
```

#### 3. 社交平台帖子（即刻 / 微博等）

```bash
# 方法 A: 通过平台工具查看
[连接命令] 'cd /root/.openclaw/jike && node jike-toolkit.mjs my-posts'
# 确认最新帖子的内容、图片数量、时间

# 方法 B: 通过帖子链接访问（如有浏览器 MCP）
# 打开 URL 查看帖子实际呈现效果

# 确认项：
# - 帖子内容完整可读
# - 配图正常显示（无乱码、无加载失败）
# - 表情/特殊字符正常
# - 链接（如有）可点击
```

#### 4. 目录结构

```bash
# 必须执行：完整目录树
[连接命令] 'find [目录路径] -type f -exec ls -la {} \;'

# 确认项：
# - 目录层级与模板一致
# - 所有期望的文件都存在
# - 文件大小合理（不是 0 字节的空文件）
```

#### 5. 代码/脚本执行结果

```bash
# 必须执行：运行验证命令
[连接命令] '[执行命令]'

# 确认项：
# - 脚本可以正常执行（无语法错误）
# - 输出结果符合预期
# - 无权限问题或依赖缺失
```

### 验证发现问题时的处理流程

```
发现问题
  ├─ 严重问题（乱码、核心功能失败、内容完全错误）
  │   ├─ 记录到 session-log（问题描述 + 截图/证据）
  │   ├─ 对应维度大幅扣分（质量 -10~15、完成度 -5~10）
  │   └─ 判断是否需要修正后重新提交
  │
  └─ 轻微问题（格式瑕疵、非核心内容遗漏）
      ├─ 记录到 session-log
      └─ 对应维度小幅扣分（-2~5）
```

## Session 管理

- 每轮培训使用独立 `--session-id`，格式：`training-[YYYY-MMDD]-R[轮次]`
- 轮次间清理 session 防止 token 膨胀
- 超过 5 个任务主动重置 session

## 培训后更新流程

培训结束后，按以下顺序更新：

1. **生成培训报告** → `references/training-history/[date]-[topic]-report.md`
2. **更新学员档案** → `data/profiles/[agent-alias].md`
   - 能力矩阵：标记新认证的能力、更新已有能力状态
   - 培训历史表：追加本次培训记录
   - 评分趋势表：追加本次各维度分数
   - 已知问题表：新发现的问题 / 更新已有问题状态
   - 关键里程碑：记录突破性进步
   - 教练备忘：更新对该学员的观察和策略建议
3. **同步到远程** → 将报告上传到 Agent 的 `training-report/` 目录
4. **过程记录归档** → `references/training-history/[date]-[topic]-session-log.md`
