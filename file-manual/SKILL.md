# 文件说明书生成器

## 核心规范（必须遵循）

1. **注释位置**：文件说明书注释块必须位于文件的起始位置（在 shebang/import 语句之后）
2. **格式严格**：必须包含 6 个章节：核心功能、输入、输出、定位、依赖、维护规则
3. **语法适配**：根据文件类型自动选择正确的注释语法
4. **维护联动**：代码变更后必须同步更新所在文件夹的 `[模块]_[文件夹名]_README.md`

## 使用流程

1. **分析文件内容**：理解代码文件的功能和职责
2. **识别数据流**：确定输入来源和输出去向
3. **分析依赖关系**：识别关键依赖和次要依赖
4. **选择注释语法**：根据文件类型选择正确的注释格式
5. **生成注释块**：使用 `template.md` 模板在文件起始位置生成规范的注释块

## 注释语法映射

| 文件类型 | 注释语法 | 示例 |
|---------|---------|------|
| Swift / Kotlin / Java / TS / JS | `//` 单行 | `// ## 核心功能` |
| Python / Shell / Bash | `#` 单行 | `# ## 核心功能` |
| HTML / CSS | `/* */` 块注释 | `/* ## 核心功能 */` |
| Go | `//` 单行 | `// ## 核心功能` |

## 模板和示例

- **模板**: 使用根目录的 `template.md` 作为基础结构
- **语法示例**: 查看 `examples/file-manual-template.md` 了解各语言的注释格式

## 维护联动清单

代码变更后需要检查的联动文档：

| 文档类型 | 路径 | 触发条件 |
|---------|------|---------|
| 文件夹 README | `[模块]_[文件夹名]_README.md` | 每次代码变更后 |
| 全局文档 | `Documentation/Basic/*` | 架构变更时 |
