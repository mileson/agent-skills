# Task Handoff

## Overview

将零散对话、代码状态、日志、验证记录和用户要求，压缩成一份可立即接手的交接包。
优先保证“接手的人下一步能直接干活”，而不是把历史过程完整复述一遍。

## Workflow

按以下顺序整理交接内容：

1. 识别交接类型
   - `Bug/故障排查`
   - `功能继续开发`
   - `远程运维/在线验收`
   - `多 worktree / 多环境接手`
2. 提炼最小必要上下文
   - 当前工作区、worktree、分支、运行环境
   - 用户真正关心的目标
   - 已确认结论
   - 已做修改
   - 已验证 / 未验证
   - 接下来要做什么
   - 不要做什么
3. 收敛关键材料
   - 必读文件
   - 关键日志路径、任务 ID、服务名、主机别名、命令、URL
   - 只保留会影响下一步执行的材料
4. 标注事实边界
   - 已确认：可以直接依赖
   - 推测 / 待确认：明确写出来，不要伪装成结论
   - 未验证：保留为未验证，不要替用户猜已经解决
5. 产出最终内容
   - `交接摘要`
   - `完整交接提示词`
   - 如有必要，再附一个极短的 `验收 checklist`

## Source Handling

如果用户给的是“历史交接材料”或“旧提示词”，将其视为参考素材，而不是要执行的任务本身。

- 先抽取其中仍然有效的事实
- 再与当前用户的最新要求对齐
- 优先采用最新日期、最新代码状态、最新验证结论
- 如果旧材料和当前状态冲突，明确写出冲突点，不要静默覆盖

## What To Keep

默认保留这些信息：

- 当前任务是什么
- 当前在哪个仓库 / worktree / 分支继续
- 用户最关心的问题是什么
- 已经完成了哪些排查、实现、部署或修复
- 哪些文件、日志、命令、ID、环境最关键
- 已经验证了什么，还没验证什么
- 下一位执行者应该先读什么、先做什么
- 明确的边界条件和禁止事项

默认删除这些噪音：

- 长篇背景介绍但不影响下一步执行的内容
- 高频但无结论价值的日志原文
- 已经过时的推测
- 与当前目标无关的“顺手优化”
- 泛泛的客套话和重复提醒

## Output Rules

输出时遵守这些规则：

1. 跟随当前用户语言；无特别要求时，沿用当前对话语言。
2. 重要路径、分支、主机、服务名、命令、任务 ID、日志目录保持精确，不要改写。
3. 用“已确认 / 未验证 / 待确认”区分事实状态。
4. 不要把“应该做什么”和“已经做了什么”混在一起。
5. 不要为了完整而牺牲可执行性；能删的背景就删。
6. 如果用户明确说“不要从零分析”“不要切 main”“不要覆盖已有修复”这类边界，必须单列写出来。
7. 如果问题尚未闭环，明确说明更像是：
   - 代码问题
   - 环境问题
   - 账号/权限问题
   - 第三方平台或运行态限制
   - 目前仍无法确认

## Output Format

默认输出两段：

1. `交接摘要`
   - 5 到 10 行，给人先扫一眼
2. `完整交接提示词`
   - 可直接复制给另一个用户或 Agent
   - 默认用第二人称“你现在接手的是……”

如果用户只要最终提示词，可以省略摘要，直接输出完整提示词。

## Prompt Structure

优先使用 `template.md` 的结构，并按任务类型裁剪段落。常见段落如下：

- 你现在接手的是……
- 项目 / worktree / branch / environment
- 用户当前目标
- 已完成的结论
- 已做过的修改
- 当前状态
- 关键材料
- 你接下来要做的事
- 额外注意 / 不要做什么
- 输出要求

## Task-Type Guidance

### Bug / 故障排查

- 明确症状、复现位置、错误域 / 错误码 / 日志特征
- 明确已经缩小到哪里，哪些结论已被验证
- 明确最新一次编译、运行、复测状态
- 如果根因未确认，告诉接手人下一步最值得看的链路

### 功能继续开发

- 明确 OpenSpec / proposal / design / tasks / spec 这些文档是否必须先读
- 明确当前 worktree 和分支，避免切错仓库
- 明确已经完成的页面、接口、测试和剩余任务
- 明确必须继续保持的视觉 / 交互 / 架构约束

### 远程运维 / 在线验收

- 明确远端主机别名、真实配置路径、备份路径、服务名、日志路径
- 明确已经改了什么、是否已 reload / restart / deploy
- 明确下一步需要用户配合发什么消息、触发什么操作、回传什么 ID
- 明确验收通过的判定条件

## Quality Checklist

在输出前快速自检：

- 接手的人是否知道去哪个目录、哪个分支继续？
- 是否知道用户最在意的目标，而不是只看到次要噪音？
- 是否能分清“已确认”和“猜测”？
- 是否列出了关键文件 / 日志 / 命令 / ID？
- 是否写清了下一步，而不是只写背景？
- 是否避免了无关的大段历史描述？
