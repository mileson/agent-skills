# 评议委员会示例

> 示例：评估 TODO CLI 工具的三种设计方案

## 场景背景

经过三种协作模式（集中调度、领域主导、对等协作）产出了三个 TODO CLI 工具方案，需要评议委员会评估并选出最佳方案。

## 团队配置

```
Chair（主席）
  ├── Product Judge（产品评委）
  ├── Tech Judge（技术评委）
  └── UX Judge（用户体验评委）
```

## 实施步骤

### Step 1: 创建团队

```
TeamCreate
├── team_name: "evaluation-committee"
└── description: "评估 TODO CLI 三种设计方案"
```

### Step 2: 生成评委

```javascript
// Chair
Task({
  name: "chair",
  subagent_type: "general-purpose",
  prompt: `
你是评议委员会主席，负责主持评估会议。

## 你的职责
1. 接收各评委的评分和理由
2. 主持讨论，解决评分分歧
3. 汇总结果，宣布最终排名
4. 输出评估报告

## 评分标准
- 产品可行性（1-10）：市场需求、用户定位、差异化
- 技术可行性（1-10）：技术栈、实现复杂度、维护成本
- 用户体验（1-10）：命令直观性、学习曲线、交互设计

## 工作流程
1. 等待三位评委汇报评分
2. 分析分歧，必要时请求补充说明
3. 汇总平均分，确定排名
4. 向 team-lead 汇报最终结果
`
})

// Product Judge
Task({
  name: "product-judge",
  subagent_type: "general-purpose",
  prompt: `
你是产品评委，专注于产品可行性评估。

## 评估维度
1. 市场需求（真实痛点？）
2. 目标用户（明确？有购买力？）
3. 竞品差异化（蓝海还是红海？）
4. 成功概率（综合考虑）

## 输出格式
向 chair 汇报：
- 方案 A 评分：X/10
- 方案 B 评分：X/10
- 方案 C 评分：X/10
- 理由：[简要说明]
`
})

// Tech Judge
Task({
  name: "tech-judge",
  subagent_type: "general-purpose",
  prompt: `
你是技术评委，专注于技术可行性评估。

## 评估维度
1. 技术栈成熟度
2. 实现复杂度
3. 性能预期
4. 长期维护成本

## 输出格式
向 chair 汇报：
- 方案 A 评分：X/10
- 方案 B 评分：X/10
- 方案 C 评分：X/10
- 理由：[简要说明]
`
})

// UX Judge
Task({
  name: "ux-judge",
  subagent_type: "general-purpose",
  prompt: `
你是用户体验评委，专注于 UX 评估。

## 评估维度
1. 命令直观性
2. 学习曲线
3. 输出清晰度
4. 交互设计

## 输出格式
向 chair 汇报：
- 方案 A 评分：X/10
- 方案 B 评分：X/10
- 方案 C 评分：X/10
- 理由：[简要说明]
`
})
```

### Step 3: 评委独立评估

各评委独立阅读方案，按评分标准打分。

### Step 4: 汇报评分

各评委向 Chair 汇报评分和理由：

```
Product Judge 汇报：
├── 方案 A (todo-guard): 8/10 - CI 门禁定位独特，Tech Lead 是真实用户
├── 方案 B (todo-cli): 5/10 - 红海市场，差异化不足
└── 方案 C (极简工具): 4/10 - 市场定位模糊

Tech Judge 汇报：
├── 方案 A: 8/10 - Rust + clap 成熟，800 行代码可控
├── 方案 B: 6/10 - 5 层架构过重，6-8 周风险高
└── 方案 C: 5/10 - 技术栈未定，存在交付风险

UX Judge 汇报：
├── 方案 A: 7/10 - 专业场景适配，符合 DevOps 习惯
├── 方案 B: 8/10 - 日常使用友好，学习曲线低
└── 方案 C: 6/10 - 极简但功能有限
```

### Step 5: 汇总排名

```
Chair 汇总：

| 方案 | 产品 | 技术 | UX | 平均分 |
|------|------|------|-----|--------|
| A    | 8    | 8    | 7   | 7.7 🥇 |
| B    | 5    | 6    | 8   | 6.3 🥈 |
| C    | 4    | 5    | 6   | 5.0 🥉 |

最终排名：A > B > C
```

### Step 6: 输出报告

Chair 向 team-lead 汇报最终评估报告。

### Step 7: 关闭团队

```
SendMessage → shutdown_request → 各评委
TeamDelete
```

## 关键要点

1. **独立性**：评委必须独立评分，避免从众效应
2. **专业性**：每个评委专注自己的领域
3. **结构化**：统一的评分标准和输出格式
4. **可追溯**：每个评分都有明确的理由
