# 对等协作模式

> 适用场景：探索性任务、创新场景、需要多角度碰撞

## 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    对等协作模式                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │

│                         ▲                                   │

│         │               │               │                   │
```

## 共享画布模板

```markdown
# 项目名称 - 共享设计画布

## 📌 问题定义（PM 填写）
_待填写_

## 🎨 设计方案（Designer 填写）
_待填写_

## 🔧 技术方案（Engineer 填写）
_待填写_

## ⚠️ 质询清单（Critic 填写）

### Q1: [质疑标题]
问题：...
影响：...
建议：...

## 🗣️ 辩论区

### 争议 1
- **Critic 质疑**：
- **Designer/Engineer 回应**：
- **PM 裁决**：
- **决策**：

## ✅ 最终决策
_待各方达成共识后填写_

## 📊 决策日志
| 时间 | 决策 | 理由 | 裁决者 |
```

## 实施步骤

### Step 1: 创建共享画布

```
Write
├── file_path: "{project}-canvas.md"
└── content: 共享画布模板
```

### Step 2: 创建对等角色

所有角色地位平等，各自负责专业领域：

| 角色 | Section | 职责 |
|------|---------|------|
| PM | 问题定义 | 定义问题、目标用户、成功指标、裁决争议 |
| Designer | 设计方案 | 命令设计、交互设计、输出格式 |
| Engineer | 技术方案 | 技术选型、架构设计、可行性评估 |
| Critic | 质询清单 | 质疑假设、竞品分析、风险识别 |

### Step 3: 并行填写

各角色并行填写自己的 section，不修改别人的内容。

### Step 4: 结构化辩论

Critic 提出质疑 → Designer/Engineer 回应 → PM 裁决

### Step 5: 共识输出

达成共识后，填写最终决策 section。

## 辩论机制

```
质疑格式：
Q1: [质疑标题]
问题：[具体问题描述]
影响：[不解决会有什么后果]
建议：[可选的改进建议]

回应格式：
- **Critic 质疑**：引用 Q1
- **Designer/Engineer 回应**：解释或修改
- **PM 裁决**：决定采用或拒绝
- **决策**：最终决定
```

## 已知限制

1. **Agent 无法直接编辑共享文件**：当前 Agent Teams 技术限制
2. **协调成本高**：需要手动同步和更新画布
3. **依赖 team-lead**：需要人工整合各角色输出

## 优缺点

| 优点 | 缺点 |
|------|------|
| 多角度碰撞 | 技术限制导致协作困难 |
| 创新性高 | 协调成本高 |
| 结构化辩论 | 产出可能不完整 |

## 最佳实践

1. **明确的 section 划分**：每个角色只编辑自己的 section
2. **编号质疑**：Critic 使用 Q1, Q2... 便于引用
3. **定期同步**：team-lead 定期读取并更新共享画布
4. **PM 裁决权**：明确 PM 有最终裁决权
