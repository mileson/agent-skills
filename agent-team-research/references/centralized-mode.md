# 集中调度模式

> 适用场景：产品决策、时间敏感任务、需要完整产出

## 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    集中调度模式                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    ┌─────────────┐                          │
│                    │ Coordinator │ ← 创建团队、整合报告      │
│                    └──────┬──────┘                          │
│                           │ 并行分配任务                     │
│         ┌─────────────────┼─────────────────┐               │
│         │                 │                 │               │
│    ┌────┴────┐       ┌────┴────┐       ┌────┴────┐          │
│    │Researcher│      │Researcher│      │Researcher│          │
│    │  产品    │       │  技术    │       │  UX     │          │

│         └─────────────────┼─────────────────┘               │

│                    ┌─────────────┐                          │

```

## 实施步骤

### Step 1: 创建团队

```
TeamCreate
├── team_name: "{project}-research"
└── description: "调研任务描述"
```

### Step 2: 生成 Researcher

根据调研维度，并行生成多个 Researcher：

| 维度 | Researcher 名称 | 职责 |
|------|-----------------|------|
| 产品 | product-researcher | 市场分析、用户需求、竞品对比 |
| 技术 | tech-researcher | 技术选型、可行性、架构方案 |
| UX | ux-researcher | 用户体验、交互设计、学习曲线 |
| 市场 | market-researcher | 市场规模、竞争格局、定价策略 |

### Step 3: 独立研究

各 Researcher 独立完成各自维度的研究，互不干扰。

### Step 4: 整合报告

Coordinator 收集所有 Researcher 的报告，整合为完整输出。

### Step 5: 关闭团队

```
SendMessage → shutdown_request → TeamDelete
```

## 优缺点

| 优点 | 缺点 |
|------|------|
| 产出完整 | 协调成本较高 |
| 质量稳定 | 可能缺乏深度碰撞 |
| 适合时间敏感任务 | Coordinator 压力大 |
| 职责清晰 | - |

## 最佳实践

1. **Researcher 数量**：建议 3-5 个，不超过 5 个
2. **并行执行**：所有 Researcher 同时启动，提高效率
3. **明确截止**：设置明确的完成时间点
4. **Coordinator 监控**：定期检查进度，及时催促
