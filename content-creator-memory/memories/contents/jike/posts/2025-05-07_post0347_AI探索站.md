# 即刻动态 347

**发布时间:** 2025-05-07  
**圈子:** AI探索站

---

最近发现越来越多创造者开始分享 Vibe Coding 的一些探索的编程思路与规范，也收获不少。

借此机会，我也将目前我探索的一套思路分享给各位创造者，希望能够对大家所有启发，由于篇幅有限，所以更多给大家一个初步的框架性认知（注意：以下为主要围绕产品需求已确定的前提下展开的 Vibe Coding 的流程部分）：

1️⃣ 新产品启动时，我将通过一整套结构化的公共 markdown 文档（见下方），使用「新版本规划提示词」保证整体需求梳理完整性，并为新版本开发奠定良好的基础（图 1）：
Documentation
	-- App-flow.md 产品流程图
	-- PRD.md 产品需求文档
	-- Frontend-guidelines.md 前端文档
	-- Backend-structure.md 后端文档
	-- Tech-stack.md 技术栈文档
	-- File-structure.md 项目文件结构文档
	-- Versions 版本目录

2️⃣ 新产品的第一次开发，我将引用 @Documentation 内所有文档，使用「拆解开发任务 Todo 提示词」，将首个版本的需求拆解成多个 @Task.md 文档，并归纳到 Versions 目录下的 Tasks 目录内，为了保证需求按版本隔离记录，方便后续管理与追溯，目录结构会演变为：
	Documentation
	-- App-flow.md 产品流程图
	-- PRD.md 产品需求文档
	-- Frontend-guidelines.md 前端文档
	-- Backend-structure.md 后端文档
	-- Tech-stack.md 技术栈文档
	-- File-structure.md 项目文件结构文档
	-- Versions 版本目录
		-- V0.1 
			-- Tasks
				-- task1.md
				-- task2.md
				...

（如果 Task 有出入，可以通过引用 Documentation 内所有文档与指定的 task.md 文档进行针对性让 AI 调整）

3️⃣ 确认拆解到任务无误后，我将按次序引用如 @task1.md 以及 @Documentation 内所有文档，使「需求分析提示词」，通过 Ask 模式让 AI 梳理并总结结果到聊天记录内以提高聊天窗口信息量密度。

4️⃣ 通过多次重复对话，确认 AI 回复已经梳理很全面了，我再通过「基于 PRD 驱动的 Task 任务执行编码提示词」让 AI 进行自动化编码并自测，最终执行的结果一方面会输出到聊天窗口内，同时也会总结到 Task.md 文档内，方便后续结构化、可视化查看与确认、留档（图 2）。

5️⃣ AI 编码完成后，在人工测试前，我会通过「自动化测试提示词」，驱动 AI 进行自动化测试。比如后端逻辑功能可用性测试通过创建 python 脚本等方式，自动运行终端命令（此处指网站类产品）达成所有测试命令都通过，即算完成目标；前端界面交互可用性测试，由 AI 生成测试用例，人工参考用例进行操作与确认（图 3），目录结构会演变为：
	Documentation
	-- App-flow.md 产品流程图
	-- PRD.md 产品需求文档
	-- Frontend-guidelines.md 前端文档
	-- Backend-structure.md 后端文档
	-- Tech-stack.md 技术栈文档
	-- File-structure.md 项目文件结构文档
	-- Versions 版本目录
		-- V0.1 
			-- Tasks
				-- task1.md
				-- task2.md
				...
			-- Tests
				-- test1.md
				-- test2.md
				...

以上就是一次「基于提示词驱动的 AI 编程」的全流程，从规划-分析-确认-编码-测试，有些细节由于篇幅有限，进行了精简，更多是面向各位创造者们提供一套思考框架。

并且这是一套正在实际开发过程中已经应用的的流程。希望能够给大家带来启发：）
